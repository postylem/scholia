# Scholia v0.2: Threading, State, and Polish

**Date:** 2026-03-12
**Status:** Design
**Author:** V + Claude
**Builds on:** v0.1 (2026-03-12)

## 1. Summary

v0.2 turns scholia from a proof-of-concept into a usable annotation tool. The core changes: true multi-turn threaded conversations, persistent read/unread and resolve/unresolve state, anchor-aligned sidebar layout, polished code blocks, and a test suite.

## 2. Threading

### 2.1 Multi-turn threads

The v0.1 data model already supports multi-turn threads (`body` is an array of messages). What changes is the browser UI: expanded cards get a reply input at the bottom, so humans can continue a thread without creating a new annotation.

- Human replies from the browser sidebar
- AI replies via `scholia reply` CLI
- All interleaved in the `body` array, each with `creator` and `created` timestamp
- No change to the JSONL format

### 2.2 Browser reply flow

When a card is expanded:
1. Thread messages are displayed chronologically
2. A text input appears at the bottom of the thread
3. Submitting sends a `reply` WebSocket message: `{type: "reply", annotation_id, body, creator: "human"}`
4. The server appends the reply to the JSONL, watchdog detects the change, broadcasts the update

## 3. Read/Unread State

### 3.1 Persistence

Read state is stored in a **separate sidecar file** (`<doc>.scholia.state.json`), not in the JSONL. Rationale: read state is UI state that changes frequently (every time you open a thread). Appending to the JSONL for every read would bloat it with noise.

State file format:
```json
{
  "urn:uuid:abc": {"lastReadAt": "2026-03-13T01:00:00+00:00"},
  "urn:uuid:def": {"lastReadAt": null}
}
```

### 3.2 Behavior

- **Mark as read:** Expanding a thread sends `mark_read` immediately (not on collapse). If new messages arrive while the thread is open, the card stays visually current — the next collapse/expand cycle will re-mark. This is simple and predictable.
- **Mark as unread:** A button on the card header clears `lastReadAt`. Useful when you want to come back to something.
- **Unread detection:** A thread is "unread" if any message in the thread has a `created` timestamp newer than `lastReadAt`, or if `lastReadAt` is null/absent. All timestamps are ISO 8601 with timezone (UTC).
- **Badge:** Unread threads show an "unread" badge on the card header. The badge text indicates the role of the last unread message (e.g., "AI reply" or "new reply").

### 3.3 Server support

New endpoints/messages:
- WS message `{type: "mark_read", annotation_id}` → server writes `lastReadAt` to state file
- WS message `{type: "mark_unread", annotation_id}` → server clears `lastReadAt`
- State file is loaded on page serve and included in the initial page data alongside comments

### 3.4 New module: `scholia/state.py`

Manages the `<doc>.scholia.state.json` sidecar file:
- `state_path(doc_path)` → Path (same pattern as `annotation_path`)
- `load_state(doc_path)` → dict (returns `{}` if file missing or corrupt JSON, logs warning on corruption)
- `mark_read(doc_path, annotation_id)` → read-modify-write: load, update `lastReadAt` to `now(UTC)`, atomic write (write to tempfile + `os.replace`)
- `mark_unread(doc_path, annotation_id)` → same pattern, set `lastReadAt` to `null`

Atomic writes via tempfile + `os.replace` prevent corruption from partial writes. For the single-user beta, this is sufficient — no file locking needed.

## 4. Resolve/Unresolve

### 4.1 Data model

Resolution is a meaningful annotation state change, so it stays in the JSONL as top-level fields on the annotation object:
- `scholia:status` field: `"open"` → `"resolved"` (or back)
- `scholia:resolvedAt` timestamp (set when resolved, cleared when unresolved)

Mechanically: `resolve()` loads the current annotation, sets the status fields, and appends the updated annotation to the JSONL (same last-write-wins-by-id deduplication used by `append_reply`). No messages are added to the `body` array — resolve/unresolve is metadata, not conversation. The `body` array contains only human/AI text messages.

### 4.2 Behavior

- **Default view:** Sidebar shows only open threads
- **Resolve:** A button on expanded cards sets status to `"resolved"`. Card disappears from default view.
- **Unresolve:** Available when viewing resolved threads. Sets status back to `"open"`.
- **View filter:** A sidebar control to "show all since [date]" reveals resolved threads whose `scholia:resolvedAt` is newer than the chosen date. This lets you review recently-resolved threads without seeing ancient history.

### 4.3 CLI

- `scholia resolve <doc> <id>` — resolve a thread
- `scholia unresolve <doc> <id>` — unresolve a thread
- `scholia list <doc> --all` — show all threads including resolved
- `scholia list <doc> --since 2026-03-12` — show all threads (including resolved) created or modified on or after this date. Date is ISO 8601 (date-only `YYYY-MM-DD` or full timestamp). Comparison is inclusive, in UTC.

### 4.4 WebSocket messages

- `{type: "resolve", annotation_id}` → server resolves, broadcasts `comments_update`
- `{type: "unresolve", annotation_id}` → server unresolves, broadcasts `comments_update`

All WS message handlers (new_comment, reply, resolve, unresolve, mark_read, mark_unread) are wrapped in try/except. On error, the server sends `{type: "error", message: "..."}` back to the originating client. Other clients are not affected.

## 5. Anchor-Aligned Sidebar Layout

### 5.1 Positioning

Cards are positioned at the Y offset of their anchor highlight in the document. This creates a visual connection between the annotation and the text it refers to.

Algorithm:
1. For each comment, find its anchor highlight element(s) in the document
2. Get the top offset of the first highlight element
3. Position the card at that Y offset in the sidebar
4. If cards overlap (anchors are close together), push later cards down, maintaining order but keeping them as close to their anchors as possible

### 5.2 Click-to-anchor interaction

Clicking a card header:
1. Expands/collapses the thread (existing behavior)
2. If expanding and the anchor is not visible in the viewport, smooth-scrolls the document to center the anchor (using `scrollIntoView({behavior: 'smooth', block: 'center'})`)
3. After scroll settles (~300ms), plays a CSS `@keyframes` pulse animation on the anchor highlight: briefly brighten the highlight background, then fade back to normal over ~600ms

### 5.3 Orphan comments

Detection is frontend-only: during `reanchorAll()`, if `TextQuoteAnchor.toRange()` returns `null` for an annotation, that annotation is an orphan for this render. Orphan status is rechecked on every doc update (the text might come back after an edit).

- Card is dimmed with a "lost anchor" indicator (strikethrough on the anchor text excerpt, muted opacity, small "anchor not found" label)
- Orphan cards are grouped at the bottom of the sidebar, below all anchored cards
- The comment thread remains fully functional (can still reply, resolve, etc.)
- Clicking an orphan card does not scroll the document (no anchor to scroll to)

## 6. Code Blocks

### 6.1 Syntax highlighting

Use Pandoc's built-in skylighting engine. Ship the highlight CSS as part of `scholia.css` (extracted from Pandoc's `pygments` theme). Pandoc outputs `<span>` elements with class names like `.kw`, `.co`, `.st`, etc.

### 6.2 Language label

A small label in the upper-right corner of each code block showing the language. Pandoc adds classes like `class="sourceCode python"` to `<pre>` elements. JS extracts the language by finding all classes on the `<pre>` that aren't `sourceCode` — the remaining class is the language name. If no language class is found, no label is shown. Language names are title-cased for display (e.g., "Python", "Haskell").

### 6.3 Copy button

A clipboard icon button in the upper-right corner of code blocks (next to the language label). Appears on hover. Copies the code content (text only, no HTML) to the clipboard. Brief "Copied!" feedback on success.

Both the language label and copy button are added by `scholia.js` post-processing Pandoc's output.

## 8. CLI Error Handling

All CLI commands should fail gracefully with clear error messages instead of Python tracebacks:

- **File not found:** `scholia start nonexistent.md` → `Error: nonexistent.md not found`
- **Annotation not found:** `scholia reply doc.md bad-id "text"` → `Error: annotation bad-id not found`
- **No comments file:** `scholia list doc.md` when no `.scholia.jsonl` exists → `No comments.` (already works, but verify)
- **Pandoc not installed:** `scholia start doc.md` without Pandoc → `Error: Pandoc not found. Install from https://pandoc.org/installing.html`
- **Port in use:** `scholia start doc.md` when port 8088 is occupied → `Error: port 8088 already in use (try --port)`
- **Invalid date:** `scholia list doc.md --since bad-date` → `Error: invalid date format, use YYYY-MM-DD`

Implementation: wrap each CLI command handler in try/except, catch known error types, print a one-line message to stderr, and exit with code 1. Unknown errors still show the traceback (useful for bug reports).
